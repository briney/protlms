## A. The Container Contract spec (becomes `docs/CONTRACT.md`)

This is the linchpin. Write it as a human-readable spec whose schemas are mirrored exactly by the Pydantic models in `src/plms/contract.py`. The doc is versioned by a single `CONTRACT_VERSION` string.

### A.1 Contract version

- `CONTRACT_VERSION = "0.1"` — a `MAJOR.MINOR` string.
- Compatibility rule (Phase 0, deliberately simple): **client and image must share the same MAJOR.** Same major + image minor `&lt;=` client minor → run. Image minor `&gt;` client minor → run with a `rich` warning (forward-compatible reads, client may not understand new fields — Pydantic ignores unknown fields by default, which we rely on). Major mismatch → hard refuse with `ContractVersionError`. Document this table explicitly in the doc.
- Rationale: a string keeps parsing trivial; the `embed`/`likelihood`/`score`/`generate` surface is reserved up front so adding `score`/`generate` later is a minor bump, not a major one.

### A.2 Internal CLI the image MUST expose

The image's `ENTRYPOINT` is the contract CLI. Subcommands:

```
&lt;entry&gt; manifest
        # prints manifest JSON to stdout, exit 0. No mounts required.

&lt;entry&gt; embed      --input /in/seqs.fasta --output /out
                   [--pooling mean|cls|none] [--layers -1] [--batch-size N] [--device cpu|cuda]
&lt;entry&gt; likelihood --input /in/seqs.fasta --output /out
                   [--batch-size N] [--device cpu|cuda]

# RESERVED — not implemented in Phase 0, but the flags are specified so the
# contract and client are forward-compatible:
&lt;entry&gt; score      --input /in/variants.csv --output /out [--device ...]
&lt;entry&gt; generate   --input /in/prompts.fasta --output /out [--num-samples N] [--temperature T]
```

Flag conventions (locked for the doc):
- `--input` is a path **inside the container** (always under `/in`, read-only).
- `--output` is a directory **inside the container** (always `/out`, writable).
- `--pooling`: `mean` and `cls` produce pooled output; `none` produces per-residue output. Manifest declares which modes the model supports.
- `--layers`: comma-separated layer indices (e.g. `-1` for last, `-1,6` for multiple). Phase 0 ESM2 supports a single layer; multi-layer is parsed but a single index is the tested path. Negative indexing allowed.
- `--device`: `cuda` only valid when the container was launched with `--gpus`. The container decides actual placement; if `cuda` requested but unavailable, it writes a structured error (see A.6).

### A.3 Manifest JSON schema → `Manifest` Pydantic model

Concrete fields (extends VISION's proposed list with explicit types):

```python
class Capability(StrEnum):
    EMBED = "embed"
    LIKELIHOOD = "likelihood"
    SCORE = "score"        # reserved
    GENERATE = "generate"  # reserved

class PoolingMode(StrEnum):
    MEAN = "mean"
    CLS = "cls"
    NONE = "none"

class Manifest(BaseModel):
    model_config = ConfigDict(extra="ignore")  # forward-compat: tolerate new fields
    contract_version: str                 # e.g. "0.1"
    name: str                             # canonical model name, e.g. "esm2_t6_8M"
    version: str                          # model/image semantic version
    description: str
    model_family: str                     # e.g. "esm2"
    capabilities: list[Capability]
    embedding_dim: int                    # e.g. 320 for esm2_t6_8M, 1280 for t33_650M
    max_sequence_length: int              # e.g. 1024
    pooling_modes: list[PoolingMode]      # e.g. [mean, cls, none]
    num_layers: int                       # transformer layer count (for --layers validation)
    min_gpu_memory_gb: float | None       # None =&gt; runs comfortably on CPU
    default_batch_size: int
    image_digest: str | None = None       # filled by registry/runner at runtime, not by image
```

Note: `image_digest` is awkward for the image to self-report (an image generally can't know its own digest reliably from inside). Decision: the **client** resolves the digest at run time (via `docker image inspect`) and records it in `Result`/run metadata, not the manifest. The manifest field stays present (Optional) for forward-compat but is authoritative when set. Flag this as the one place VISION's field list and reality diverge.

### A.4 `result.json` schema → `Result` Pydantic model

The container writes `/out/result.json` on success. The client reads it to discover what was produced (rather than globbing `/out`).

```python
class OutputArtifact(BaseModel):
    path: str                  # relative to /out, e.g. "embeddings.npz" or "per_residue/seq1.npy"
    kind: str                  # "pooled_embeddings" | "per_residue_embeddings" | "likelihoods_csv"
    record_ids: list[str] | None = None   # for per-residue: which record this file is
    shape: list[int] | None = None        # array shape if applicable
    dtype: str | None = None               # e.g. "float32"

class Result(BaseModel):
    model_config = ConfigDict(extra="ignore")
    contract_version: str
    capability: Capability
    model_name: str
    n_input_records: int
    n_output_records: int      # may differ if some sequences were skipped
    artifacts: list[OutputArtifact]
    warnings: list[str] = []   # e.g. "3 sequences truncated to max_sequence_length"
    params: dict[str, str] = {}  # echo of pooling/layers/etc. for reproducibility
```

### A.5 I/O convention (exact filenames)

- **Mounts:** `-v &lt;host_in&gt;:/in:ro` and `-v &lt;host_out&gt;:/out:rw`. Input file path passed as `/in/&lt;filename&gt;`.
- **Input:** FASTA for `embed`/`likelihood`. (`score` reserved → CSV with columns `record_id,mutation` or `record_id,sequence`; documented but not implemented.)
- **Embed — pooled** (`--pooling mean|cls`): single file `/out/embeddings.npz`. Keys = record IDs (sanitized), each value a `(embedding_dim,)` float32 array. `Result.artifacts` has one `pooled_embeddings` entry with `shape=[n_records, embedding_dim]` (logical shape) and the npz key list.
- **Embed — per-residue** (`--pooling none`): one `.npy` per record under `/out/per_residue/&lt;sanitized_id&gt;.npy`, each `(L_i, embedding_dim)` float32. One `per_residue_embeddings` artifact entry per file. Document the ID-sanitization rule (replace non-`[A-Za-z0-9._-]` with `_`, dedupe collisions with `__N` suffix) because FASTA headers are messy.
- **Likelihood — CSV:** `/out/likelihoods.csv` with header columns:
  `record_id,seq_len,pseudo_log_likelihood,mean_pseudo_log_likelihood,pseudo_perplexity`
  - `pseudo_log_likelihood`: sum over positions of log P(true residue | rest) under masked-marginal.
  - `mean_pseudo_log_likelihood`: that sum / `seq_len` (length-normalized — the usually-comparable quantity).
  - `pseudo_perplexity`: `exp(-mean_pseudo_log_likelihood)`.
  - One artifact entry, `kind="likelihoods_csv"`.

### A.6 Error contract

- Exit code `0` = success and `/out/result.json` present.
- Non-zero exit = failure. The container writes a **single JSON object to stderr** matching:

```python
class ContainerError(BaseModel):
    model_config = ConfigDict(extra="ignore")
    contract_version: str | None = None
    error_type: str       # e.g. "InvalidInput" | "UnsupportedCapability" |
                          #      "SequenceTooLong" | "DeviceUnavailable" | "InternalError"
    message: str
    details: dict[str, str] = {}
```

- Client behavior: on non-zero exit, attempt to parse the **last** JSON object on stderr into `ContainerError`. If parseable → raise `ContainerExecutionError` carrying the structured fields (rendered nicely via `rich`). If not parseable → raise `ContainerExecutionError` with the raw stderr tail (last ~50 lines) so a crash/segfault is still legible. Always log the full `docker run` argv and exit code first.

### A.7 What CONTRACT.md contains (sections)

1. Versioning &amp; compatibility table. 2. Subcommand reference (all four, reserved marked). 3. Manifest schema. 4. Result schema. 5. I/O &amp; filename conventions. 6. Error schema. 7. A worked example: `manifest` output JSON + `embed --pooling mean` producing `embeddings.npz` + `result.json` for a 3-sequence FASTA. The doc and the Pydantic models in `contract.py` are the two sides of the same coin; keep them edited together.

### B.4 `src/plms/runner.py`

The Runner seam (locked decision #2). A `Protocol` interface + a subprocess implementation. Returns a structured result; does not parse domain outputs (that's `io.py`).

```python
@dataclass(frozen=True)
class RunSpec:
    image: str
    command: list[str]                 # contract subcommand + flags, e.g. ["embed","--input","/in/x.fasta",...]
    input_dir: Path                     # host path mounted ro at /in
    output_dir: Path                    # host path mounted rw at /out
    use_gpu: bool = False
    extra_env: dict[str, str] = field(default_factory=dict)

@dataclass(frozen=True)
class RunResult:
    exit_code: int
    stdout: str
    stderr: str
    argv: list[str]                     # the exact command, for logging/tests

class Runner(Protocol):
    def run(self, spec: RunSpec) -&gt; RunResult: ...

class SubprocessDockerRunner:
    def __init__(self, docker_executable: str = "docker") -&gt; None: ...
    def run(self, spec: RunSpec) -&gt; RunResult: ...
    def manifest(self, image: str) -&gt; str: ...   # convenience: `docker run --rm IMAGE manifest`, returns stdout
```

`build_argv(spec)` — a module-level pure function (the testable core, kept &lt;50 lines):
```
[docker, "run", "--rm",
 ("--gpus","all")?,                       # only when spec.use_gpu
 "-v", f"{spec.input_dir}:/in:ro",
 "-v", f"{spec.output_dir}:/out:rw",
 (for k,v in env: "-e", f"{k}={v}"),
 spec.image,
 *spec.command]
```

`run()`: log argv at INFO via `logging` (global convention: stdlib logging in library code, no `print`); `subprocess.run(argv, capture_output=True, text=True, check=False)` — **never `shell=True`**, always a list (global "what not to do"). Wrap `FileNotFoundError`/`OSError` (docker missing) → `RunnerError`. Return `RunResult`. It does **not** raise on non-zero exit — the caller (`Model`) interprets exit codes + parses `ContainerError`. Keep error-parsing helper here or in `contract`; put `parse_container_error(stderr) -&gt; ContainerError | None` in `contract.py` (pure) and let `Model` call it.

Depends on: `subprocess`, `logging`, `contract` (for nothing heavy — just types). No `docker-py`. The `Protocol` is what lets a `DockerSdkRunner` drop in later without touching `Model`.

### B.6 `src/plms/models.py` — where `plms.load(name)` lives

`plms.load(name)` is a thin module-level function re-exported in `__init__.py` that returns a `Model`. The `Model` object is the user-facing unified interface: it owns a `ModelEntry`, a lazily-fetched `Manifest`, and a `Runner`.

```python
def load(name: str, *, runner: Runner | None = None,
         registry: Registry | None = None) -&gt; Model:
    """Resolve a model name to a ready-to-use Model (fetches &amp; validates the manifest)."""

class Model:
    def __init__(self, entry: ModelEntry, runner: Runner, manifest: Manifest) -&gt; None: ...

    @property
    def manifest(self) -&gt; Manifest: ...

    def embed(self, fasta: str | Path, *, pooling: str = "mean",
              layers: Sequence[int] = (-1,), output_dir: Path | None = None,
              use_gpu: bool = False, batch_size: int | None = None) -&gt; EmbeddingResult: ...

    def likelihood(self, fasta: str | Path, *, output_dir: Path | None = None,
                   use_gpu: bool = False, batch_size: int | None = None) -&gt; LikelihoodResult: ...
```

`load()` flow: `Registry.resolve(name)` → `runner.manifest(entry.image)` → `Manifest.model_validate_json(stdout)` → `check_contract_compatibility(manifest.contract_version)` → construct `Model`. If image absent locally, `docker run` fails → map to `ImageNotFoundError` with a "build it from containers/esm2" hint.

`embed()`/`likelihood()` flow (shared private `_run_capability` helper, &lt;50 lines):
1. Validate capability in `manifest.capabilities` → else `CapabilityNotSupportedError`.
2. Validate `pooling` against `manifest.pooling_modes` → else `InvalidRequestError`.
3. `read_fasta` for record count + length pre-check against `manifest.max_sequence_length` (warn, don't fail — container truncates and reports in `Result.warnings`).
4. `io.stage_inputs` → temp `/in` + `/out`.
5. Build `RunSpec` (command via a small `_build_command` per capability) → `runner.run`.
6. On non-zero exit → `parse_container_error` → raise `ContainerExecutionError`.
7. On success → `io.read_result` + the right loader → return a result object.

Result wrappers (lightweight dataclasses, give users typed access without re-reading files):
```python
@dataclass
class EmbeddingResult:
    result: Result
    output_dir: Path
    pooling: str
    def pooled(self) -&gt; dict[str, np.ndarray]: ...        # lazy load
    def per_residue(self) -&gt; dict[str, np.ndarray]: ...

@dataclass
class LikelihoodResult:
    result: Result
    output_dir: Path
    def rows(self) -&gt; list[dict]: ...
```

Output-dir semantics: if `output_dir` given, results are copied out of the temp dir to it and the wrapper points there; if `None`, the wrapper holds a tempdir kept alive for the object's lifetime (document this; simplest correct behavior for Phase 0). The CLI always passes `-o`, so the persistent path is the common case.

Depends on: `registry`, `runner`, `io`, `contract`, `exceptions`. This is the integration module — it's the only one that touches all the others, by design.

## C. The ESM2 container (`containers/esm2/`)

Build context: `Dockerfile`, `entrypoint.py` (the contract CLI), and a small README documenting build args. Excluded from the wheel (already true since wheel only packages `src/plms`).

### C.1 Library choice: HuggingFace `transformers` (`facebook/esm2_*`), not `fair-esm`

Reasons: (1) HF has first-class masked-LM heads (`EsmForMaskedLM`) which gives pseudo-log-likelihood almost for free via masked-marginals, plus `EsmModel` for hidden states — both capabilities from one dependency. (2) HF model+tokenizer download is a clean, cacheable build step (`AutoModel.from_pretrained(ckpt)`), making weight-baking trivial. (3) `fair-esm` is lighter but its API for per-layer reps and PLL is more manual and its checkpoint naming differs. (4) ESM-C and ProGen2 (Phase 1) are also HF-friendly, so standardizing on `transformers` now reduces per-container divergence. The build arg maps `esm2_t6_8M` → `facebook/esm2_t6_8M_UR50D` via a small dict in the entrypoint.

### C.2 Dockerfile design

- Base: `pytorch/pytorch:&lt;pinned&gt;-cuda12.x-cudnn-runtime` (CUDA already present, torch pinned, works on the A6000; falls back to CPU when run without `--gpus`). Pin a digest for CI reproducibility per global Docker conventions (use tag locally, digest in CI).
- Install: `transformers`, `numpy` (pinned). No `plms` package inside — the container is independent of the client.
- **Build arg + baked weights** (locked decision #4, and VISION's reproducibility principle):
  ```dockerfile
  ARG ESM2_CHECKPOINT=esm2_t6_8M
  ENV ESM2_CHECKPOINT=${ESM2_CHECKPOINT}
  COPY entrypoint.py /app/entrypoint.py
  # bake weights at build time so runtime needs no network:
  RUN python /app/entrypoint.py _prefetch     # downloads + caches the HF checkpoint into the image
  ENTRYPOINT ["python", "/app/entrypoint.py"]
  ```
  `_prefetch` is a hidden subcommand that calls `from_pretrained` to populate the HF cache layer. This bakes weights → reproducible, offline runtime (set `HF_HUB_OFFLINE=1` at runtime to enforce). Build the demo image with `--build-arg ESM2_CHECKPOINT=esm2_t6_8M`.

### C.3 entrypoint.py — implements the contract

A single-file argparse CLI (no typer needed inside the container; keep its deps minimal) exposing `manifest`, `embed`, `likelihood`, plus hidden `_prefetch`. Each subcommand small.

- `manifest`: print a hard-coded-per-checkpoint `Manifest`-shaped JSON. `embedding_dim`/`num_layers` come from a per-checkpoint table (320/6 layers for t6_8M; 1280/33 for t33_650M) or read from `model.config`. `capabilities=["embed","likelihood"]`, `pooling_modes=["mean","cls","none"]`, `max_sequence_length=1024`, `contract_version="0.1"`.
- `embed`:
  - Read FASTA from `--input`. Tokenize in batches (`default_batch_size`, overridable). `model.eval()`, `torch.no_grad()` (global PyTorch conventions), `torch.amp` autocast on CUDA.
  - Hidden states from `EsmModel(..., output_hidden_states=True)`; select `--layers` index.
  - Pooling: `mean` → average over residue positions excluding special tokens (BOS/EOS/pad) using the attention mask. `cls` → the BOS token representation. `none` → strip special tokens, keep `(L_i, D)`.
  - Write per A.5: pooled → `embeddings.npz` (`np.savez`); per-residue → `per_residue/&lt;id&gt;.npy`. Write `result.json`. Shape comments on tensor ops (`# (B, L, D) -&gt; (B, D)` for mean-pool).
- `likelihood`:
  - `EsmForMaskedLM`. Pseudo-log-likelihood via masked-marginals: for each position, mask it, run forward, take log-softmax logit of the true residue. Batch the masked variants per sequence (cap batch by `--batch-size`). Exclude special tokens from the sum.
  - Emit `likelihoods.csv` per A.5 columns + `result.json`.
  - Document the cost: masked-marginal PLL is O(L) forward passes per sequence — fine for tiny test seqs; note it as a perf caveat (a future fast-path could use a single unmasked forward "naive PLL").
- Error handling: top-level try/except writes a `ContainerError` JSON to stderr and `sys.exit(1)`; map known cases (`SequenceTooLong`, `DeviceUnavailable`) explicitly.
- Truncation: sequences &gt; `max_sequence_length` → truncate, append a `Result.warnings` note (don't hard-fail) — matches the client's warn-don't-fail pre-check.

## Risks, sequencing concerns, and VISION ambiguities to flag

- **`image_digest` in the manifest** is not reliably self-knowable from inside a container. Decision: client resolves it at runtime (`docker image inspect`) and records it in run metadata/`Result`; the manifest field stays optional. This is the one concrete divergence from VISION's field list — call it out in CONTRACT.md.
- **Packaged registry location.** VISION shows top-level `configs/`, but a registry that ships in the wheel must live inside the package (`src/plms/_data/models.yaml`) for `importlib.resources`. Recommend the in-package location; keep top-level `configs/` for repo-level (non-packaged) config if/when needed. Minor VISION ambiguity.
- **Masked-marginal PLL cost** is O(L) forward passes/sequence. Fine for tiny test seqs and the demo; document it and note the future fast-path. Don't let the integration test use long sequences.
- **`--layers` multi-layer output format** is under-specified by VISION. Phase 0: support a single layer on the tested path; if multiple layers are requested for per-residue, namespace files as `per_residue/&lt;id&gt;__layer&lt;L&gt;.npy`. Document and keep the multi-layer path lightly tested.
- **No-`--gpus` CPU correctness.** The pytorch base image must run on CPU when launched without `--gpus`. Ensure entrypoint's `--device` defaults to `cuda if torch.cuda.is_available() else cpu` and the client only passes `cuda` when `use_gpu=True`. Test the CPU path in CI-like conditions (the integration test should run CPU by default; GPU is an extra assertion on the A6000).
- **Contract/`contract.py` drift** is the central maintenance risk — the doc and the Pydantic models are two encodings of one spec. Mitigation: a CONTRACT.md "worked example" JSON that's also loaded as a fixture in `tests/test_contract.py`, so a doc/code mismatch fails a test.
- **`docker build` time in integration tests** (pulling the pytorch base + baking weights) can be minutes on first run. Gate behind `@pytest.mark.slow` + an env flag, session-scope the build, and never run it in the standard CI matrix.
